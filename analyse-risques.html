Risques légaux

Risques juridiques & réglementaires

⚖️ ORIAS — Activité réglementée assurance
CRITIQUE
Tout professionnel qui négocie, souscrit ou présente une assurance pour le compte de tiers doit être immatriculé ORIAS (Art. L. 511-1 Code des assurances). Sans cela : exercice illégal d'une activité réglementée, sanctions pénales.

En V1 : les recommandations d'assurance doivent être des redirections vers un courtier ORIAS tiers, jamais une intermédiation directe.
Mitigation V1 : Redirection vers comparateur agréé ou courtier ORIAS partenaire. Aucun acte d'intermédiation directe. Mention "En partenariat avec [courtier ORIAS immatriculé]" obligatoire. V2 : partenaire ORIAS. V3 seulement : ORIAS propre.
🏥 HDS — Données de santé
CRITIQUE
Le stockage de données de santé (carte vitale, ordonnances, bilans médicaux) requiert la certification HDS (Hébergeur de Données de Santé) — Art. L. 1111-8 CSP. Coût estimé : 20 000–80 000€ et 6 à 12 mois de délai (audit ANSSI).
Mitigation V1 : Exclure toutes les données de santé du coffre-fort V1. La catégorie "Santé" dans le schéma est créée mais désactivée (feature flag = false). Le schéma health est prévu en V3 uniquement.
🪪 Données d'identité biométriques
CRITIQUE
Le stockage de CNI recto/verso et passeport pages biographiques est soumis à la délibération CNIL 2023 sur les données biométriques et d'identité. Requiert un audit CNIL préalable et des mesures techniques renforcées.
Mitigation V1 : Documents d'identité exclus de la V1 (feature flag = false sur catégorie "identite"). Prévu en V2 après audit CNIL. La catégorie existe dans le schéma mais la UI est grisée/masquée.
📊 RGPD — Données personnelles sensibles
MAJEUR
Documents fiscaux, contrats, IBAN, quittances : toutes des données personnelles Art. 4 RGPD. Nécessite consentement explicite (Art. 6.1.a), politique de conservation claire, DPO désigné si traitement à grande échelle.
Mitigation : Table rgpd_consent_logs (append-only, Art. 7). Soft-delete cascade + anonymisation PII J+30. Export JSON/CSV en 1 clic (portabilité). Registre des traitements Art. 30. Chiffrement AES-256 documents. DPA avec hébergeur Supabase.
🔀 Conflits d'intérêts — DGCCRF
MAJEUR
Le modèle de commissions sur les recommandations (affilié télécom, courtier assurance) crée un risque de conflit d'intérêt si non déclaré. Pratique commerciale trompeuse sous Art. L. 121-1 Code de la consommation.
Mitigation : Contrainte DB commission_must_be_disclosed — impossible d'insérer sans commission_disclosed = TRUE. Mention obligatoire sur chaque carte de recommandation : "Commission perçue si souscription : ~X€". Architecture déclarative by design.
⚖️ Mandats SSP — Horodatage légal
MAJEUR
Les mandats sous Art. 1301 Code Civil doivent être horodatés et révocables à tout moment. En V1, l'horodatage simple (created_at TIMESTAMPTZ + document_hash SHA-256) peut être contesté. La preuve de date est limitée.
Mitigation V1 : created_at précis + SHA-256 du PDF. V2 : Prestataire TSA (Trusted Service Authority) qualifié eIDAS ou intégration CertEurope/DocuSign pour horodatage qualifié. Mandat max 1 an, non auto-renouvelable (contrainte DB).
🏛 Concurrence étatique — France Connect+
MODÉRÉ
France Connect+, Espace Santé CNAM, Mon Espace Agent FP sont des initiatives étatiques avec accès privilégié aux données. Risque que l'État développe un "super-coffre-fort" concurrent avec accès direct aux bases opérateurs.
Mitigation : Différenciation par la délégation familiale (que l'État ne fera jamais), l'UX, et le segment couple/aidant. En V3 : intégration France Connect+ comme source de données plutôt que concurrent.
📱 Loi Chatel — Délais de notification
MODÉRÉ
La Loi Chatel impose à l'opérateur de notifier l'utilisateur avant la reconduction tacite. L'application ne peut pas garantir que l'opérateur a bien notifié — elle s'appuie sur les dates saisies par l'utilisateur.
Mitigation : Disclaimer explicite "Dates basées sur votre saisie — vérifiez votre contrat". La tacit_renewal_date n'est jamais auto-calculée. Le trigger d'alerte est basé sur la date saisie, pas sur une API opérateur.
Risques techniques

Risques base de données & architecture

RisqueNiveauDescriptionMitigation
FK circulaire households ↔ profiles Majeur profiles.household_id → households.id et households.created_by → profiles.id. Chicken-and-egg à l'INSERT. Créer household FIRST, puis UPDATE profiles.household_id dans une fonction SECURITY DEFINER atomique.
Race condition Stripe webhooks Majeur Stripe garantit at-least-once delivery. Double webhook → double mise à jour de plan_tier. Ajouter stripe_event_id TEXT UNIQUE sur user_subscriptions. Dédupliquer au niveau applicatif avant d'appeler le trigger.
RLS multi-politiques perf Modéré 3-4 politiques PERMISSIVE par table (documents, subscriptions) → évaluées en OR → risque de seq scan si mal indexé. Toutes les conditions RLS sur colonnes indexées. Index partiels WHERE deleted_at IS NULL. Monitoring EXPLAIN ANALYZE sur les requêtes critiques.
Quota storage non-natif Modéré Supabase Storage n'a pas de limite native par utilisateur. Un user peut dépasser son quota si la vérification applicative est bypassed. Vérification v_user_storage < plan.storage_quota_bytes AVANT l'upload côté serveur (Edge Function). Jamais côté client.
Mandats sans auto-expiration Modéré Les mandats expirés restent avec revoked_at IS NULL si le pg_cron n'a pas tourné. Un mandataire pourrait accéder à des données post-expiration si la RLS ne vérifie pas expiry_date. La RLS sur documents vérifie expiry_date >= CURRENT_DATE. Le pg_cron alerte J-30 mais ne révoque pas automatiquement (décision légale : révocation toujours par le mandant).
RGPD Art. 17 — PII résiduels Modéré Soft-delete rend les données invisibles via RLS mais elles restent physiquement en base. audit_logs et rgpd_consent_logs contiennent des PII. Job pg_cron J+30 post-soft-delete : anonymise colonnes PII (email → "[SUPPRIMÉ]", ip_address → NULL). audit_logs conservés pour obligation légale (mention dans CGU/politique de conservation).
Multi-devises V3 Faible v_user_monthly_cost assume tout en EUR. Ajout de devises casse la vue. Ajout d'un filtre WHERE currency = 'EUR' à la vue dès V1. Colonne currency CHAR(3) déjà présente sur subscriptions. Migration V3 : table exchange_rates.
Risques business

Risques commerciaux & marché

🏆 Concurrents existants
MAJEUR
Pappers, Tricount, Bankin' (Budget Insight), Papiers (La Banque Postale) couvrent des segments adjacents. Aucun ne combine coffre-fort + délégation familiale + alertes reconduction, mais ils ont des bases d'utilisateurs établies.
Différenciation : Aucun concurrent ne cible explicitement le segment "enfant adulte qui gère les abonnements de ses parents vieillissants". La délégation familiale SSP est un moat légal difficile à copier rapidement.
💸 Dépendance revenus affiliation
MAJEUR
Les revenus d'affiliation (commissions télécom/streaming) dépendent des programmes partenaires qui peuvent changer leurs termes ou taux à tout moment. Amazon Associates a coupé des taux de 10% à 3% en 2020.
Mitigation : Diversifier les sources de revenus. SaaS subscriptions comme revenus principaux et stables. Commissions affiliation = upside, pas baseline. Contractualiser les partenariats clés (SFR, Free, etc.).
📉 Churn plan couple en cas de séparation
MODÉRÉ
Le plan couple est lié à deux comptes. Une séparation peut provoquer un churn de 2 utilisateurs simultanément. Si mal géré (perte de documents partagés), risque de réputation.
Mitigation : Table life_events avec type 'separation' → workflow de séparation guidé (qui garde quoi, export des documents partagés). La séparation doit être une feature premium, pas un abandon brutal.
🔒 Sécurité — Fuite de documents
FAIBLE (si implémenté)
Les documents (contrats, IBAN, baux) ont une forte valeur pour des acteurs malveillants. Une fuite = fin de confiance et potentiellement RGPD Art. 83 (amendes jusqu'à 4% CA mondial).
Mitigation : Chiffrement AES-256 au repos via Supabase Vault. TLS 1.3 en transit. RLS stricte par user_id. Buckets privés (jamais public). Audit logs sur tous les accès documents. 2FA obligatoire sur plan Famille+.
